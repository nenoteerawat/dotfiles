# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository Purpose

Personal dotfiles for macOS, forked from [craftzdog/dotfiles-public](https://github.com/craftzdog/dotfiles-public). The repo is checked out into `~/.ghq/github.com/nenoteerawat/dotfiles`; top-level files are intended to live at `$HOME` and `.config/*` at `$XDG_CONFIG_HOME` (`~/.config`). There is **no build step** — once a config is symlinked, editing it here edits the live config. A file only affects the live config once it is actually symlinked into place, so don't assume an edit is live: verify the target is a symlink back into the repo (e.g. `readlink ~/.zshrc`).

Bootstrap is handled by `./install.sh` — an idempotent installer that installs every referenced tool via Homebrew/ghq/npm/mise, clones fzf-tab + TPM, and sets up **zsh** (it intentionally skips the fish config). Run `./install.sh --help` for tiers/flags. It is **Apple-Silicon-only** (the config hardcodes `/opt/homebrew`).

Linking is **per authored file**, not per directory: the installer enumerates `git ls-files` and symlinks each tracked file to its mirror under `$HOME`, keeping `~/.config/*` as **real directories**. This is deliberate — tool-generated files (`lazy-lock.json`, `lazyvim.json`, `~/.config/tmux/plugins/`, Mason, lazygit `state.yml`) then live in the real config dirs and never pollute the repo. Adding a new tracked config file means re-running `./install.sh` to link it. **Exception:** `.scripts/` is linked as a **whole-directory symlink** (`HOME_DIRS` in `install.sh`, not the per-file `CONFIG_DIRS` loop) — safe because nothing tool-generated is ever written into `~/.scripts/`, and it means new scripts appear there immediately **without** re-running `install.sh`.

## Common Commands

### Applying changes

- **Zsh** (`.zshrc`) — `source ~/.zshrc` or open a new shell.
- **tmux** (`.config/tmux/tmux.conf`) — inside tmux: `prefix + r` (prefix is `Ctrl+t`). First-time plugin install: `prefix + I` (TPM).
- **Neovim** (`.config/nvim/`) — restart `nvim`. Plugins are managed by LazyVim: `:Lazy sync`, `:Lazy update`, `:Mason` for LSP/tools.
- **Fish** (`.config/fish/`) — open a new fish shell; `exec fish` to reload in place.
- **macOS defaults** — `./.macos` (re-applies all system tweaks; logout/restart may be needed for some).
- **TouchID for sudo** — `./update_sudo_tid.sh`.

### Commits

Use conventional commits via cz-git (configured in `.czrc` → uses `cz-git` adapter):

- `git cz` — interactive commit (Commitizen).
- In lazygit: press `C` on the files view to launch the same flow (`.config/lazygit/config.yml`).
- Manual commits should follow `<type>(<scope>): <description>` (types: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `ci`).
- **Prerequisites:** `git cz` / lazygit `C` need `commitizen` + `cz-git` installed globally (npm). Several `.gitconfig` git aliases (`git a`, `git df`, `git find`) shell out to `peco`, and `git open` needs `hub` — they fail silently if those binaries are absent.

### CI (GitHub Actions)

`.github/workflows/` runs two `anthropics/claude-code-action@beta` workflows (both auth via the `CLAUDE_CODE_OAUTH_TOKEN` secret): `claude.yml` responds to `@claude` mentions in issue/PR comments, and `claude-code-review.yml` auto-reviews every opened/synchronized PR.

### Syncing with upstream

Upstream is `craftzdog/dotfiles-public`. The README's "Syncing with upstream" section is the source of truth. **Note:** the `craftzdog` remote is not configured by default — currently the only remote is `origin`, and upstream content is mirrored as the `origin/craftzdog` branch. So either add the remote first (one-time) or sync from `origin/craftzdog`:

```bash
git remote add craftzdog https://github.com/craftzdog/dotfiles-public.git   # one-time, if missing
git fetch craftzdog
git log master..craftzdog/master --oneline   # review (or master..origin/craftzdog if not added)
git merge craftzdog/master --no-commit       # merge with chance to cherry-pick
git diff --staged                            # discard upstream changes you don't want
git commit -m "feat: sync with upstream craftzdog/dotfiles-public"
```

Keep personal customizations (custom `.zshrc` functions, `.gitconfig` aliases, ghq sparo helpers) isolated so upstream merges of Neovim/tmux configs stay clean.

## Architecture & Layout

### Top-level files map directly into `$HOME`

- `.zshrc` — primary shell config. Sources `zsh-autosuggestions`, `zsh-syntax-highlighting`, and `zsh-history-substring-search` from Homebrew (`$(brew --prefix)/share/...`); loads fzf keybindings via `source <(fzf --zsh)`; and registers many `source <(<tool> completion zsh)` lines. `NODE_EXTRA_CA_CERTS` points at `~/.ssh/gateway-ca-cloudflare.pem` (the Cloudflare gateway cert — migrated from the old Zscaler cert). Order matters: `compinit` (and `bashcompinit`) run **once, early** — right after the `fpath` entries (brew `site-functions` + Docker completions) are set, but **before** the `source <(<tool> completion zsh)` lines — so `compdef` is defined before any completion registers itself. Each tool-completion line is guarded with `command -v <tool>` so an uninstalled tool is skipped, not an error. Note `fzf-tab` is **not** a Homebrew tool — it is sourced from a local ghq clone (`~/.ghq/github.com/Aloxaf/fzf-tab/...`) near the very end, after `starship init` (which is correct: fzf-tab must load after `compinit`).
- `.gitconfig` — includes `[url ...] insteadOf` rewrites that route the work GitHub orgs (`pttep-pcl`, `pttep-fusionsol`) to the **work-account SSH host alias** `github.com-pttep` (which maps to `~/.ssh/pttep_gh_id` via `.ssh/config`). For each org **three** URL spellings are rewritten — `https://github.com/<org>/`, the scp-style `git@github.com:<org>/`, and `ssh://git@github.com/<org>/` — so `git clone` *and* `ghq get` work with any form. The `ssh://` form matters specifically because **ghq normalizes scp-style URLs into `ssh://` before cloning**; without it `ghq get git@github.com:pttep-pcl/…` would fall through to the default `github.com` host (the personal key) and fail with a misleading "Repository not found". Because ghq derives the local path from the *original* URL (before these rewrites), repos still land at the clean `~/.ghq/github.com/<org>/<repo>`. Keep this in mind when debugging clone behavior.
- `.ssh/config` — the **only** file tracked under `.ssh/` (`.gitignore` has `.ssh/*` + `!.ssh/config`, so private keys/`known_hosts`/certs are never committed). Defines two GitHub identities: `Host github.com` → `personal_gh_id` (default, personal account) and `Host github.com-pttep` → `pttep_gh_id` (work account, for the pttep orgs). Both use `IdentitiesOnly yes`. SSH can't pick a key by repo path, so org→key routing is done in `.gitconfig` (above), not here. Linked into `~/.ssh/config` by `install.sh`.
- `.macos` — a **minimal, snapshot-style** `defaults` script (~55 lines, **no `sudo`** — there are no system-level writes). Philosophy: keep macOS at its defaults and encode **only this machine's deliberate deltas**. It deliberately **replaced** the inherited craftzdog/mathiasbynens 978-line kitchen-sink (Safari/Mail/Dock opinions plus sections for Twitter, Transmission, Opera, SizeUp, … apps that aren't installed) — recover that from git history if ever needed. **Three deltas are scripted** (each snapshotted from the live machine): (1) **language/region** — `AppleLocale=en_TH`, `AppleLanguages=(en-TH, th-TH)` (English/Thai, Thailand; en_TH already implies metric/Celsius; needs a **logout** to fully apply); (2) **trackpad tap-to-click** — `Clicking=true` on both `com.apple.AppleMultitouchTrackpad` and `com.apple.driver.AppleBluetoothMultitouch.trackpad`; (3) **Spotlight's launcher hotkey** — rebinds "Show Spotlight search" (`symbolichotkeys` id 64) from its `⌘ Space` default to `⌥ Space` (mask `524288`), freeing `⌘ Space` for Alfred (the write is followed by `killall cfprefsd` + the private `activateSettings -u`). **Tahoe caveat:** the write records the right value but on macOS 26 the running Spotlight may not re-bind (`⌘ Space` keeps opening Spotlight even after a reboot, with cfprefsd/domain/MDM/input-source causes all ruled out) — make it live by toggling it **once** in the GUI (System Settings → Keyboard → Keyboard Shortcuts → Spotlight → record `⌥ Space` for "Show Spotlight search"), after which the scripted line just keeps the state. **Alfred's own `⌘ Space` hotkey is GUI-only** and can't be scripted at all (machine-local, non-synced, inside the `Alfred.alfredpreferences` bundle `preferences/local/<machine-hash>/` — no `defaults`/AppleScript/Powerpack-CLI hook); afterwards confirm `⌘ Space` isn't also grabbed by Input Sources (`⌃ Space` default) or Dictation. **To extend it:** change a setting in System Settings, read it with `defaults read <domain> <key>`, add a line in the matching section — but verify the write actually *live-applies* (some Tahoe settings, like the Spotlight hotkey, only fully apply via the GUI), and keep the file a faithful mirror of real deltas, not a kitchen-sink.
- `update_sudo_tid.sh` — idempotent (early-exits if `pam_tid.so` is already in `sudo_local`). Rewrites `/etc/pam.d/sudo_local` in one `tee`, but first **backs up** any existing file to `sudo_local.bak` (no longer a blind clobber). The `pam_reattach` line (needed for TouchID inside tmux/screen) is only written when `/opt/homebrew/lib/pam/pam_reattach.so` actually exists — referencing a missing PAM module would break `sudo` entirely with "unable to initialize PAM"; `pam_tid.so` always ships with macOS. After writing it runs a safety-net `sudo -n true` and aborts with restore instructions if PAM no longer initializes. Still hardcodes the **Apple-Silicon** Homebrew path (Intel would need `/usr/local`).

### `.config/` mirrors `$XDG_CONFIG_HOME`

- `.config/nvim/` — LazyVim-based config.
  - `init.lua` → `lua/config/lazy.lua` is the LazyVim bootstrap. Plugin **extras** are listed there (TS, JSON, Go, Rust, Docker, YAML, Helm, Tailwind, SQL, ESLint, Prettier, mini-hipatterns, GitHub). Personal plugin specs live in `lua/plugins/` (`coding.lua`, `colorscheme.lua`, `editor.lua`, `lsp.lua`, `treesitter.lua`, `ui.lua`) and are auto-imported by `{ import = "plugins" }`. Colorscheme (`tokyonight`, style `night`) is selected via the LazyVim opts in `lazy.lua` *and* specced in `plugins/colorscheme.lua` — keep the two in agreement. The spec remaps tokyonight to the macOS 26 Terminal **"Clear Dark"** palette via `on_colors` (so nvim matches Terminal.app + the tmux theme) and keeps `transparent = true` to inherit the terminal's translucent background. Note tokyonight snapshots derived roles (`error`/`warning`/`hint`/`git`/`diff`/`border_highlight`/`rainbow`/`terminal`) *before* `on_colors` runs, so those are re-set inside `on_colors` — don't drop them.
  - `lua/craftzdog/` holds three inherited, live utilities required from `lua/config/keymaps.lua`: `discipline.lua` (rate-limits `hjkl` spam), `hsl.lua` (`<leader>r` hex→HSL), `lsp.lua` (inlay-hints / autoformat toggles). The sibling `plugins.lua` is **dead packer cruft** left over from upstream — never imported (this repo uses lazy.nvim) and safe to delete.
  - `dev = { path = "~/.ghq/github.com" }` in `lazy.lua` lets local clones under ghq override remote plugin sources — useful when hacking on a plugin locally.
  - Claude Code integration is provided by `coder/claudecode.nvim` in `plugins/coding.lua` under `<leader>a*` keymaps.
- `.config/tmux/` — split across `tmux.conf` (entrypoint), `theme.conf`, `statusline.conf`, `utility.conf` (lazygit popup on `prefix+g`, Claude Code popup on `prefix+y` that re-attaches a per-directory `claude-<hash>` session — relies on `md5sum`), and `macos.conf` (loaded only on Darwin). Prefix is `Ctrl+t`. Plugins via TPM (`~/.config/tmux/plugins/tpm`): `tmux-pain-control` + `craftzdog/tmux-claude-session-manager` — **all pane navigation/resize/split bindings (`hjkl`, `HJKL`, `|`, `-`) come from pain-control, not the conf** (the local versions are commented out). After adding a plugin, install it with `prefix + I`. The status bar (`theme.conf`/`statusline.conf`) is themed to the macOS Terminal **"Clear Dark"** palette with `bg=default` to inherit the terminal's translucency (matching the nvim tokyonight remap). Gotcha: `statusline.conf` is sourced after `theme.conf` and overrides its status-bar styling, so editing status colors in `theme.conf` has no visible effect.
- `.config/fish/` — `config.fish` dispatches to `config-osx.fish` / `config-linux.fish` / `config-windows.fish` (empty) based on `uname`, then optionally sources `config-local.fish` if present (machine-specific, untracked overrides). `functions/fzf_change_directory.fish` is the fish twin of the zsh `fzf_change_directory` widget — keep their search roots in sync. Fish uses **starship** (`config.fish` runs `starship init fish`); the old `conf.d/tide.fish` Tide config was removed. Note `install.sh` does **not** set up fish — it's a zsh-only bootstrap — so the fish tree is now effectively unmaintained relative to the primary zsh config.
- `.config/lazygit/config.yml` — adds a `C` custom command that runs `git cz c` (needs cz-git installed globally), and binds `openDiffTool` to `<c-e>`.
- `.config/mise/config.toml` (pins bun/ghq/node/python/ruby/rust to `latest` — note **ghq is managed by mise here**, not only Homebrew) and `.config/ghostty/config` are single-file tool configs. `.config/powershell/` is the Windows shell+prompt setup — two files: `user_profile.ps1` (posh-git, PSReadLine, PSFzf, Terminal-Icons, aliases) driven by the `takuya.omp.json` oh-my-posh theme.

### `.scripts/`

`~/.scripts` is a **whole-directory symlink** into the repo (`install.sh` → `HOME_DIRS`, see the linking note above) and is prepended to `$PATH` in `.zshrc`, so every script below is a **bare command from any repo** (e.g. `cc-auth work`, not `~/.scripts/cc-auth work`). Reload with `source ~/.zshrc` after a fresh clone. Dropping a new script into `.scripts/` makes it available immediately — no `install.sh` re-run.

- `ide` — run inside an existing tmux window to create a 5-pane IDE layout (left editor pane + right column split). Invoke as `ide` (on `$PATH`).
- `cc-credits-refresh [work]` — fetches month-to-date org API spend from the Anthropic Cost API and writes the non-secret cache `~/.claude/cache/credits-<acct>.json`. **The only file that reads the Admin key** (see the status line section). The statusline spawns it in the background when the cache is stale; can also be run on a cron/launchd timer.
- `cc-work-limit` — Claude Code **hook** (not a CLI; registered in the tracked `.claude/settings.json` for `UserPromptSubmit` + `PreToolUse` matcher `*`) enforcing the **$100/day work-account cap**: warns at ≥80% of `WORK_DAILY_BUDGET`, **blocks prompts and denies tool calls at ≥100%** (the PreToolUse deny is what stops a long *running* turn mid-flight). Reads the local ledger the statusline writes (see the `day` segment bullet); a session counts as work-billed iff `ANTHROPIC_API_KEY` is in its environment (Claude Code injects the cc-auth-stamped `env` block into every subprocess, hooks included; fallback: key *presence* checked in the repo's `settings.local.json` via jq). Personal/subscription sessions always exit 0 untouched; every failure path degrades to "allow". Budget comes from the same stamp (default 100 if unset; `0` disables enforcement). Also prunes ledger days older than 14 days — kept here so the statusline stays write-only fast.
- `cc-auth [work|personal|status|strip-global]` — switches which **credentials Claude Code authenticates with**; the statusline's spend segments (`day`, `acct`) follow this same auth state automatically, so this is the single work/personal switch (the old `cc-acct` display toggle + `~/.claude/statusline-account` file were removed). "Option B" per-repo model: the global `~/.claude/settings.local.json` is kept auth-free so Claude.ai **subscription** is the default everywhere, and the work API credentials (key + AWS external-Anthropic gateway `ANTHROPIC_BASE_URL` + `anthropic-workspace-id` header — all three, since the gateway needs the base URL/header too, not just the key) are stamped into a **single repo's** `.claude/settings.local.json` on demand. `work`/`personal` toggle the current repo (resolved via `git rev-parse --show-toplevel`); `status` reports the repo + global state; `strip-global` is the one-time migration that moved the block out of the global file (backup at `settings.local.json.bak`). The secret lives **only** in `~/.claude/work-auth.json` (a chmod-600 JSON object of the work `env` entries, **never committed**, lives in `$HOME`); `work` refuses to stamp it while it still contains `REPLACE` placeholders, and keeps each per-repo `settings.local.json` out of git via that repo's **local** `.git/info/exclude` (never edits the tracked `.gitignore`). `install.sh` seeds a 600 `work-auth.json` template on fresh machines. Because cc-auth stamps **whatever keys are in `work-auth.json`** (read dynamically — the block can grow without touching the script), non-secret work config rides the same stamp: `WORK_DAILY_BUDGET` (the $100/day cap used by the statusline `day` segment and the `cc-work-limit` hook) is just one more entry there. Repos stamped **before** a new entry existed need one `cc-auth work` re-run to pick it up (stamping merges, so re-runs are safe). Remember Claude Code only reads auth at session start, so toggling affects **new** sessions, not the running one; run `/login` once so a subscription session exists for the personal default. **Expected warning:** starting Claude Code in a work repo prints *"Both claude.ai and ANTHROPIC_API_KEY set · auth may not work as expected"* — this is **harmless**. Claude Code credential precedence puts `ANTHROPIC_API_KEY` (#3) above subscription OAuth (#6), so the key wins and the gateway call uses it (the subscription OAuth can't reach a custom `ANTHROPIC_BASE_URL` anyway). Approve the key once when prompted to make it deliberate; confirm with `/status`. Do **not** `/logout` (that clears the subscription in every other repo) or set the key globally.

### Local AI coding implementor (`--with-localai`, Ollama + OpenCode)

An **architect → implementor** split for offline/zero-cost coding on this Apple-Silicon machine (M-series, 48 GB): **Claude Code is the architect** (plans, writes a per-task spec + the repo's `AGENTS.md`) and a **local open model is the implementor** (edits the repo autonomously, diff left uncommitted for review). Installed via `./install.sh --with-localai` (gated — the model pull is ~25 GB, so it never runs on a default install). Three moving parts:

- **Runtime — Ollama (native, Homebrew).** `brew services start ollama`. **Important, easily-mistaken fact:** for the GGUF models used here Ollama runs on the **llama.cpp Metal** backend, **not** its MLX engine (the MLX engine is gated to NVFP4/MLX-specific builds these models don't ship). The real footgun is **context**: Ollama's default is **4 K and it silently truncates** overflow, quietly breaking agent loops. Setting `OLLAMA_CONTEXT_LENGTH=65536` via `launchctl setenv` proved **unreliable** (the brew-services daemon capped it to 32768 anyway), so the robust fix is to **bake `num_ctx` into the model via a Modelfile** (see the model bullet) — that reliably yields the full 65536 (verified with `ollama ps`, which must also show **100% GPU**). Still set `OLLAMA_FLASH_ATTENTION=1` (server-level) and keep KV cache at **F16** (Q8 at most, never Q4 — coding is quant-sensitive). *Persistence:* `launchctl setenv` and the loaded model don't survive a reboot; re-`ollama run` reloads on demand. If you ever want true MLX throughput, that means leaving Ollama for **LM Studio** headless (`lms server start`), not `mlx_lm.server` (officially not for production).
- **Model — Devstral is the reliable default; Qwen3.6-35B-A3B is the benchmark challenger.** `devstral-small-2:24b` (Mistral, Apache-2.0, 24B dense, SWE-bench Verified 68.0 — **Mistral-official**, independently confirmed 2026-07) has proven native tool-calling and an official Ollama tag → it's the default in `opencode.json`, pulled at **q8_0** (~25 GB, tag `devstral-small-2:24b-instruct-2512-q8_0`) and **rebuilt under the clean name `devstral-small-2:24b` via a Modelfile that bakes in `num_ctx 65536` + `temperature 0.2` + `top_p 0.95`** (`ollama create devstral-small-2:24b -f Modelfile`). Baking the context into the model is what actually guarantees it — the `OLLAMA_CONTEXT_LENGTH` env alone got capped to 32768 in testing. `Qwen3.6-35B-A3B` (Apache-2.0, 35B/3B-active MoE, self-reported SWE-bench 73.4 — **failed independent verification** in a 2026-07 adversarial deep-research pass, treat as unproven; faster + more headroom, ~30 GB total at Q6_K per Unsloth's official table) is stronger *on paper* but its community GGUF has **reported tool-call/hallucination issues** at Q6 (unsloth GGUF discussion #15) — partly offset by Unsloth's Qwen3.6 uploads documenting improved nested-object tool-call parsing and developer-role support naming **OpenCode** specifically — so it's **not** pulled by default. **Getting it is a two-step gotcha (verified 2026-07 on Ollama 0.30.x):** `ollama pull hf.co/unsloth/Qwen3.6-35B-A3B-GGUF:Q6_K` downloads all ~29 GB of blobs but then **fails the final import with `Error: 400`** — an hf.co-manifest bug, *not* the weights (so `ollama cp` from the pulled tag can't work; the tag never registers). The path that works is to build **from the already-downloaded blob**: find the ~27 GB `sha256-…` file in `~/.ollama/models/blobs/`, point a Modelfile `FROM` it with `PARAMETER num_ctx 65536` (+ `temperature 0.2`/`top_p 0.95`, same bake as Devstral), then `ollama create qwen3.6-35b-a3b -f Modelfile` — that sidesteps the 400 *and* bakes in the context (verified: loads 100% GPU at 65536 ctx, coherent output). Then **benchmark both on a real task from your repo** before promoting it to the `opencode.json` default — the two SWE-bench figures use different scaffolds and are **not a head-to-head**, so the local benchmark is the deciding test. **Verified upstream gotchas (2026-07, confirmed against primary sources):** (a) Ollama's Go runner **silently discards `repeat_penalty`/`presence_penalty`/`frequency_penalty`** — accepted by the API, no penalty sampling implemented (ollama#14493) — so penalty lines in Modelfiles are no-ops and repetition loops in long agent sessions can't be fixed that way (the baked `temperature`/`top_p` are separate samplers and still apply); (b) the same issue documents **Qwen3.5 tool calls mis-wired to Hermes-style JSON instead of the XML format the model was trained on** (open on master) — verify qwen3.6 tool calls parse cleanly in OpenCode before trusting benchmark results; (c) the hf.co import failure also manifests upstream as `unknown model architecture: qwen35moe` (ollama#15834, other versions) — the FROM-blob build above (or the official ollama.com `qwen3.6:35b` tag, quant unspecified) sidesteps every variant; **never benchmark via a direct hf.co pull**. The same 2026-07 research pass found **no verified better sub-40 GB open coding model** than this pair — every roundup alternative (Kimi K2.6, DeepSeek-V4-Flash, Qwen3-Coder-Next, GLM-5, …) failed 3-vote verification, several with fabricated scores, so trust only primary sources when re-evaluating.
- **Harness — OpenCode (autonomous agent CLI), not Aider.** `brew install opencode`. Config is the **tracked, symlinked** `.config/opencode/opencode.json` (added to `install.sh` `CONFIG_DIRS`; defines the `ollama` provider via `@ai-sdk/openai-compatible` at `http://localhost:11434/v1`, both models, and `permission: {edit: allow, bash: ask, webfetch: deny}`). OpenCode is model-agnostic, does native tool-calling, and **does not auto-commit** (session snapshots → the diff sits in the working tree for `git diff` review — matching the review-before-commit intent). Run headless: `opencode run -m ollama/devstral-small-2:24b "$(cat .task-spec.md)"` — **edit-only tasks need no bypass flag** because `opencode.json` sets `permission.edit: allow` (verified: OpenCode reads/edits the file and leaves it uncommitted). There is **no `--auto` flag** in 1.17.x (the current OpenCode docs use that name for the same behavior); the unattended flag is `--dangerously-skip-permissions`, and — verified on 1.17.10 (`opencode run --help`) — it **auto-approves only permissions that are *not explicitly denied***, so config `deny` rules always hold. The local-implementor flow (bullet below) needs **no bypass flag at all**: its `implement` agent profile has no "ask" rules (bash is explicit allow/deny throughout), so headless runs are prompt-free by construction — prefer that over the flag. Add `--format json` to script it, or `opencode serve &` + `opencode run --attach …` to kill cold-start latency. The **handoff** uses the open **`AGENTS.md`** standard (auto-loaded by OpenCode) for durable repo conventions + the Claude-authored per-task spec piped in as the prompt — no bespoke wrapper. *(An earlier draft used Aider + a custom `~/.scripts/implement` wrapper + an ad-hoc `SPEC.md`; that was replaced because Aider is a file-scoped pair-programmer, not an autonomous implementor, and OpenCode+AGENTS.md is the idiomatic 2026 stack.)*
- **Using it (the architect → implementor flow).** The day-to-day loop, fully local + zero-cost after step 1: **(1) Design** — in Claude Code (the architect), plan the change and write it to a per-task spec (e.g. `.task-spec.md`); put durable, repo-wide conventions in the target repo's root **`AGENTS.md`** (OpenCode auto-loads it, so it need not be repeated each task). **(2) Implement** — `brew services start ollama` (once per boot), then from the target repo run `opencode run "$(cat .task-spec.md)"` — **no `-m` flag** (it defaults to Devstral via `opencode.json`). OpenCode + Devstral read the spec + `AGENTS.md`, edit the files, and leave the diff **uncommitted** in the working tree (it **never auto-commits**). **(3) Review** — `git diff`; good → commit (`git cz`); wrong → refine the spec and re-run, or fix by hand. Notes: the first run after ~5 min idle pays a one-time ~8 s model load (back-to-back runs start instantly); Qwen only ever loads if you explicitly pass `-m ollama/qwen3.6-35b-a3b`; check health anytime with `ollama ps` (want **100% GPU**, **65536** context). Measured on the M5 Max (48 GB): prefill ~860 tok/s, decode ~21 tok/s — i.e. big specs ingest cheaply, you only pay the slow rate for tokens the model *writes*.
- **`local-implementor` — the flow packaged as a Claude Code skill (hands-free, phrase-triggered).** Saying **"yes let devstral implement"** / **"yes let qwen implement"** (or plain "implement") in a Claude Code chat after a plan is agreed runs the whole loop: Claude writes `.task-spec.md` → runs `opencode run --agent implement "$(cat .task-spec.md)"` in the background (no bypass flag; **the model is picked from your words** — devstral is the default/no `-m`, "qwen" appends `-m ollama/qwen3.6-35b-a3b` after checking the model is actually built) → reviews `git diff` against the spec + runs the spec's verify command → on problems overwrites the spec with a `# Fix round N of 3` fix list and re-runs — **max 3 rounds**, then trivial residuals are fixed by Claude directly. The spec is deleted at the end and the diff is **always left uncommitted** for `git cz`. The skill is **user-level and machine-local by choice**: it lives at `~/.claude/skills/local-implementor/SKILL.md` as a real file (NOT tracked in this repo — deliberate, so only the `opencode.json` guardrails are repo-managed); on a fresh machine, recreate it by hand or copy it over. **Guardrails:** the `implement` agent in `opencode.json` sets bash to default-`allow` with an always-enforced **deny-list** (`sudo`, `rm -rf`, `git commit/push/reset/clean`, `curl`/`wget`/`ssh`/`scp`, `npm publish`, and the *dangerous subcommands only* of `kubectl`/`gcloud`/`aws`/`gh` — deletes/drains/applies/merges — read-only `kubectl get`/`gh pr view`/`aws s3 ls` stay allowed; **last matching pattern wins**, so the `*` catch-all stays first). No "ask" rules anywhere in the profile → headless runs are prompt-free without any bypass flag; denials don't block the model — it's told and continues. It's harm reduction, not a sandbox (a script written then executed isn't caught) — the layered backstops are: OpenCode never commits, edits are git-recoverable, Claude reviews every diff. Hands-free on the Claude side needs a `permissions.allow` block with `Bash(opencode run --agent implement:*)`, `Write(.task-spec.md)`, `Bash(rm .task-spec.md)` — it lives in the **machine-local, gitignored `~/.claude/settings.local.json`** (already set up on this machine; a machine-local skill gets a machine-local grant, and the tracked `settings.json` stays free of it). The user's explicit allow rule *is* the authorization for unattended runs — Claude editing its own permission grants without an explicit user request is blocked by design.

### Claude Code status line (`.claude/statusline.sh`)

- `.claude/statusline.sh` — the Claude Code status line ("Beacon"). It's one of **two** tracked files under `.claude/` (with `settings.json`): `.gitignore` ignores all local Claude state with `.claude/*` and re-includes just those two (`!.claude/statusline.sh`, `!.claude/settings.json`) — same idiom as the `.ssh/*` + `!.ssh/config` block. `install.sh` links both via `HOME_FILES` → `~/.claude/statusline.sh` and `~/.claude/settings.json`.
- **Activation lives in `~/.claude/settings.json`, which is now tracked + symlinked.** The status line only renders when `settings.json` has a `statusLine` block (`{"type":"command","command":"~/.claude/statusline.sh","padding":0}`). Because `settings.json` is tracked at `.claude/settings.json` and symlinked by `install.sh`, theme/effort/`enabledPlugins`/hooks/statusLine all sync to a fresh machine automatically — no hand-editing. It's portable (hooks use `$HOME`, statusLine uses `~`); put anything machine-local or secret in `~/.claude/settings.local.json`, which stays gitignored. Gotcha: since the live file is a symlink into the repo, changing settings via `/config` edits the repo file directly (it shows up as a git diff). If a Claude Code update ever rewrites it as a real file and breaks the symlink, re-run `install.sh` to re-link.
- **Design:** flat, foreground-only truecolor on the inherited (translucent) background — matches the tmux `bg=default` + nvim `transparent` "Clear Dark" philosophy; **no** background fills or powerline glyphs. Reads the status JSON on stdin via `jq` (hard dependency; degrades to a plain `Claude` if absent). Deliberately shows only what starship/tmux **don't**: model name (the one bright-blue accent), ghq-shortened `<org>/<repo>` cwd, branch + dirty `*` + ahead/behind, a yellow **work** badge for pttep repos (remote `github.com-pttep`/`pttep-pcl`/`pttep-fusionsol` or the org path segment), live context-window `%` (escalates yellow≥70→red≥90), session `$cost`, and lines `+/-`.
- **Performance gotcha:** Claude Code runs this on every render (debounced 300ms, in-flight runs cancelled), so it uses git **plumbing only** (`rev-parse`, one `rev-list` for ahead/behind, `--no-optional-locks status --porcelain`) and never the network. Keep it fast if you extend it. Test with `echo '{...}' | ~/.claude/statusline.sh`.
- **Work daily-cap segment (`day $X/$100`) + local spend ledger.** For **work-billed** sessions (detected by `ANTHROPIC_API_KEY` in the inherited environment — Claude Code injects the cc-auth-stamped `env` block into subprocesses; jq-presence fallback on the repo's `settings.local.json`) the statusline both **records** and **renders** today's spend: each render writes the session's cumulative `cost.total_cost_usd` to `~/.claude/cache/work-day/<local YYYY-MM-DD>/<session_id>` (monotonic writes, one file per session → concurrent sessions never race; sum with awk over the files *directly*, never `cat |` — files carry no guaranteed trailing newline and piping once merged `5`+`96` into `596`). The segment shows the day total vs `WORK_DAILY_BUDGET` (default 100, `0` = show plain figure, no cap), yellow ≥80% → bright-red ≥100% — deliberately the same thresholds as the `cc-work-limit` hook, which enforces the cap from the same ledger. Unlike `acct` this needs **no Admin key and no network**, resets at **local** midnight, and only counts Claude Code usage on this machine.
- **Org API spend segment (`acct …`).** For the work account it also shows month-to-date Anthropic API spend vs a budget plus today's spend (`acct 62% $312/$500 · today $12`, the pct escalating yellow≥70→red≥90). Critical facts: (1) this needs an **Admin API key** (`sk-ant-admin01-…`) — the Cost API (`/v1/organizations/cost_report`) rejects a regular `sk-ant-api…` key, and is unavailable on individual (non-org) accounts; (2) there is **no API for prepaid credit *balance*** — only spend, so the budget number is yours (set `WORK_MONTHLY_BUDGET`); (3) the statusline **never reads the key** — it only reads the non-secret cache written by `~/.scripts/cc-credits-refresh`, which sources the key from `~/.claude/statusline-credits.env` (chmod 600, **never committed**, lives in `$HOME` not the repo) and passes it to curl via a stdin config (`-K -`) so it never appears in `ps`/argv; (4) today's figure is **not** a second fetch — the month-to-date call already uses `bucket_width=1d`, so the refresher just picks today's UTC bucket out of that same response and caches it as `spent_today`, which the statusline renders only when present (old caches without it fall back to `$0` until the next refresh). States: `acct 62% $…` (ok) · `acct ⚠` (no/invalid Admin key) · `acct ✗` (API error) · `acct …` (fetching). The segment is shown only in **work-billed** sessions — same auth-state detection as the `day` segment, driven by `cc-auth` (personal is individual → no Cost API; there is no separate display toggle). Cost data lags ~5 min and the endpoint allows ~1 poll/min, so the cache TTL is 10 min and refreshes happen in a detached background process — never inline. `install.sh` seeds the cache dir + a 600 credential template on fresh machines.

### Personal `.zshrc` helpers worth preserving

These are not in upstream and should survive merges:

- `cdz <key>` — `cdz git` fuzzy-jumps into a ghq-managed repo; `cdz pttep|desk|load` jumps into `~/pttep`, `~/Desktop`, `~/Downloads`; any other arg is treated as a base path.
- `ghq-sparo <url> [profile]` — wraps `sparo clone` with the right ghq directory layout for partial checkouts.
- `fzf_change_directory` (`Ctrl+F`) — combined fzf picker over `$HOME/.config`, this dotfiles repo, ghq repos, and specific pttep GitLab groups. New search roots should be added here and mirrored in the fish version.
- Aliases declared as **functions** (`g`, `ll`, `lla`, `llt`, `llta`, `cdz`, `vim`, `vi`, `gcl`) — done this way for lazygit compatibility; don't convert them to `alias` without checking lazygit still launches editors correctly.

### Bootstrap order in `.zshrc` (important when adding lines)

1. Homebrew shellenv → PATH for everything else.
2. Plugin sources (autosuggest, syntax-highlight, history-substring).
3. `fzf --zsh` keybindings.
4. NVM.
5. **Completion system:** set `fpath` (brew `site-functions` + `~/.docker/completions`), then `compinit`, then `bashcompinit`. This must come **before** anything that calls `compdef`.
6. Tool completions via `source <(<tool> completion zsh)` — each guarded with `command -v <tool>`; plus the gitlab-ci-local `compdef` block. (These rely on step 5 having defined `compdef`.)
7. mise activate, custom functions, and `bindkey` lines.
8. `starship init zsh` — must stay near the end.
9. `fzf-tab` plugin source (correctly after `compinit`).
10. Terraform/Terragrunt/Vault `complete -C` lines (use the `bashcompinit` from step 5; guarded with `[ -x ... ]`).

When adding a new `source <(tool completion zsh)` line, put it in step 6 (after `compinit`) and guard it — never before step 5, or `compdef` won't exist yet.
